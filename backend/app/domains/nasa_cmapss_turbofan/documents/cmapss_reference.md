# NASA C-MAPSS and FD001 Reference

NASA publishes the C-MAPSS turbofan benchmark as run-to-failure simulation data for prognostics work. This file curates source-backed baseline facts used by the local demo and keeps the important FD001 semantics in plain text Markdown.

## Why NASA simulated degradation data

The DASHlink publication on prognostic algorithm development explains why simulated degradation data matters. Prognostics needs data that contains both fault signatures and fault evolution indexed over time or operational cycles. Real fleets often lack this combination: systems that fail may not have had enough instrumentation, while systems that are monitored are usually repaired before they actually run to failure. NASA positions high-fidelity simulation as practical way to create repeatable degradation histories for training, validation, and verification.

## What C-MAPSS is

C-MAPSS stands for Commercial Modular Aero-Propulsion System Simulation. NASA describes it as a tool for simulating a realistic large commercial turbofan engine. The model lets users choose operational profiles, controllers, environmental conditions, and thrust levels, then inspect many output variables that behave like sensor channels. NASA technical material also describes the C-MAPSS engine as a two-spool high-bypass turbofan with five rotating turbomachinery modules.

## How degradation is represented

NASA documentation repeatedly ties engine degradation to changes in module efficiency and flow-related parameters. The DASHlink publication says users can tune efficiency and flow parameters to simulate fault modes. The 2012 NASA technical memorandum says gradual health degradation is modeled with two health modifiers per module: efficiency and flow capacity. This matters for RAG because terms like compressor degradation, efficiency loss, and flow-capacity loss all point back to same health-modeling idea.

## Why sensor channels exist in dataset

The dataset page says several sensor channels were recorded to characterize fault evolution. The publication page adds that C-MAPSS exposes an extensive list of output variables that simulate sensor channels. In other words, benchmark is not only final label table. It is meant to pair degradation progression with measured or simulated operating signals so prognostic methods can reason about state before failure.

## PHM08 challenge corpus structure

NASA states that the public PHM08 challenge data was generated by running engine degradation simulation with C-MAPSS. Four different sets were simulated under different combinations of operational conditions and fault modes. This is key context when users ask what FD001 means: FD001 is one member of a broader family, not whole benchmark by itself.

## What FD001 specifically represents

Within the standard C-MAPSS benchmark usage, FD001 is commonly treated as the simplest of the four benchmark subsets, with one operating condition and one fault mode. In this repository, the domain manifest and signal payload interpret that pinned case as one sea-level operating condition with high-pressure-compressor degradation. That wording is repo interpretation for this local benchmark pack unless a more specific NASA source is cited alongside it.

## Remaining useful life meaning

For the C-MAPSS benchmark, remaining useful life means how many operational cycles remain after the last observed cycle until the engine reaches failure. NASA's real-flight run-to-failure paper formalizes prognostics as multivariate time series paired with RUL targets for each unit. In practical FD001 use, the model does not observe failure directly in the test rows. It observes a truncated history, then predicts cycles still left before end of life.

## Observed-cycle versus failure-cycle semantics

Observed-cycle count and failure-cycle count are not same number. The observed cycle is last row currently visible to the diagnostician or model. The failure cycle is later point where simulated engine reaches end of life. Therefore, expected failure cycle is derived as:

- last observed cycle
- plus published remaining useful life label
- equals projected failure cycle

This semantic split is central to auditability because the test data intentionally stops before failure while the label file tells how much life remained beyond that stopping point.

## FD001 unit 1 example used in this repository

This repository pins one micro-sample from FD001 test unit 1. Local manifest and signal files say last observed cycle is 31 and NASA ground-truth RUL label for that unit is 112 cycles. Under benchmark semantics, that implies expected failure cycle 143. This example is not claim about all engines in FD001. It is a repo-local deterministic example chosen so the demo can replay one concrete health-gate decision against published NASA label semantics.

## Limits of benchmark interpretation

C-MAPSS is a simulated benchmark, not direct maintenance record from a fielded airline fleet. NASA built it to support prognostics method development, not to stand in for every operational policy decision by itself. The benchmark gives structured degradation trajectories, operating settings, and simulated sensor outputs. Any operational recommendation built on top of it should state clearly where NASA source facts end and where repo-specific demo framing begins.
